# ADR-2026-03-23: parse CLI PDF 어댑터 전략

- 날짜: 2026-03-23
- 관련 이슈: AID-175, AID-165, AID-155, AID-154
- 상태: 승인 제안
- 결정 유형: 가역

## 배경

현재 `scripts/parse`는 이미지와 PDF를 모두 `hyocr`로 보낸다. 최근 PoC 결과는 다음을 보여준다.

- `odl-fast`는 디지털 텍스트 기반 PDF에서 빠르고 구조화 품질이 좋다.
- `odl-fast`는 스캔형 PDF에서 본문 OCR 대체재가 아니다.
- `odl-hybrid`는 현재 pyenv 3.11.6 기본 경로에서는 `_lzma` 부재로 실패하지만, `_lzma`가 있는 3.10+ 격리 venv에서는 재현 가능하다.
- 현행 `hyocr` 실패는 PDF 특성보다 `127.0.0.1:8080` GLM 백엔드 부재에서 재현된다.

따라서 "기본 엔진 즉시 교체"는 품질 이득보다 회귀 위험이 크다.

## 결정

`parse`의 PDF 경로는 v1에서 "명시적 선택형 어댑터"로 확장한다.

- 기본 PDF 경로는 유지한다. `parse sample.pdf`는 계속 `hyocr`로 보낸다.
- 새 옵션 `--pdf-engine`을 도입한다.
- v1 지원 값은 `hyocr`, `odl-fast`다.
- `odl-fast`는 옵트인 경로로만 제공한다.
- `odl-hybrid`는 기본 경로에서 제외한다. 전용 3.11+ 격리 venv + health check 통과 조건에서만 후속 검토한다.
- `auto` 라우팅은 v1 범위에서 제외한다. 문자 수/폰트 유무/이미지형 여부 3신호 기준선 확정 전에는 도입하지 않는다.

즉, v1 목표는 "기본 교체"가 아니라 "안전한 선택지 추가"다.

## 근거

1. 현재 `parse`는 매우 얇은 라우터다. 여기에 선택지를 추가하는 것은 저비용이다.
2. 기본 동작을 유지하면 기존 PDF 사용자 경로의 회귀를 피할 수 있다.
3. `odl-fast`는 보고서형 PDF에서 가치가 분명하므로, 명시적 opt-in만으로도 즉시 실사용이 가능하다.
4. `odl-hybrid`는 "기본 pyenv 경로 불가 + 격리 venv 조건부 가능" 상태이므로, 기본값으로 넣으면 운영 재현성이 떨어진다.
5. `auto`는 3신호 기준선 없이 도입하면 오탐으로 기본 라우팅 회귀를 일으킬 수 있다.
6. `hyocr`는 GLM 백엔드 가용성 점검 없이 기본 경로로 두면 실패 원인 파악이 늦어진다.

## v1 인터페이스

예상 CLI는 아래와 같다.

```bash
parse report.pdf --pdf-engine odl-fast
parse form.pdf --pdf-engine hyocr
parse report.pdf                    # 기존과 동일하게 hyocr
```

동작 규칙:

- 비 PDF 입력은 현행 라우팅을 유지한다.
- PDF 입력에서만 `--pdf-engine`을 해석한다.
- `--pdf-engine odl-fast`일 때만 `opendataloader-pdf` CLI를 호출한다.
- `opendataloader-pdf`가 PATH에 없으면 명확한 에러 메시지와 non-zero 종료 코드를 반환한다.
- PDF 외 입력에서 `--pdf-engine`이 들어오면 무시하거나 경고하되, v1에서는 동작을 깨지 않는다.

## 구현 경계

스파이더맨(개발)에게 넘길 구현 범위:

- `scripts/parse`에 최소 인자 파싱 추가
- `run_odl_fast()` 경로 추가
- 도움말/사용법 업데이트
- 기본 동작 회귀 없이 `--pdf-engine odl-fast`만 추가

헐크(R&D)에게 넘길 후속 범위:

- `auto` 라우팅을 위한 3신호 임계값 튜닝(문자 수/폰트/이미지형)
- `odl-hybrid` 전용 런타임 health check 자동화
- 샘플 PDF군 확대에 따른 분기 오탐률 측정

호크아이(QA)에게 넘길 검증 범위:

- `hyocr` 기본 경로 회귀 없음 확인
- `odl-fast` 옵트인 경로 정상 동작 확인
- PATH 부재, 비지원 플래그, 비 PDF 입력 혼합 시나리오 확인
- `--pdf-engine auto`, `--pdf-engine odl-hybrid` 입력 시 가드레일 에러 메시지 확인

## 기술 이전 기준 (AID-175)

아래는 개발팀 후속 구현 시 고정 기준선으로 사용한다.

1. `odl-hybrid`는 기본 경로로 가정하지 않는다.
2. `odl-hybrid`를 다룰 때는 전용 3.11+ 격리 venv에서 `_lzma`/버전/`/health`를 먼저 확인한다.
3. `auto`는 최소 3신호(문자 수, 폰트 유무, 이미지형 여부)를 함께 보지 않으면 도입하지 않는다.
4. `hyocr` 장애는 PDF 품질 문제와 GLM 백엔드 가용성 문제를 분리해 진단한다.

## 보류한 대안

### 대안 1. 기본 PDF 엔진을 `odl-fast`로 교체

기각. 스캔형 PDF에서 즉시 품질 손실이 발생한다.

### 대안 2. `auto`를 바로 기본값으로 도입

기각. 현재 저장소에는 PDF 특성 감지 로직이 없고, 오탐 시 운영 회귀가 크다.

### 대안 3. `odl-hybrid`를 함께 넣고 health check로 우회

보류. 환경 재현성부터 확보되어야 한다.

## 결과

이 결정은 가역이다.

- `odl-fast` 품질과 운영성이 충분히 검증되면 기본값 승격이 가능하다.
- `auto`가 안정화되면 후속 ADR로 기본 라우팅 변경을 검토할 수 있다.
- `odl-hybrid` 격리 환경 검증이 끝나면 세 번째 엔진으로 편입할 수 있다.

현재 시점의 최적 전략은 "기본값 유지 + 명시적 선택 추가"다.
