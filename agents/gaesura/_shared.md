## 세션 시작 (필수 — 모든 에이전트)
1. `.paperclip/agents/learnings/{본인파일}.md`를 읽어라. 과거 실수가 기록되어 있다. 같은 실수를 반복하지 마라.
2. 항목에 `관련: [[에이전트]]` 크로스레퍼런스가 있으면, 해당 에이전트의 learnings도 참조하라.

## 이슈 제목 규칙
- **한글 친화적**. 영어 기술 용어 나열 금지. 1초 만에 파악 가능하게.
- 기술 용어는 괄호 안에 보조로 넣는다.

## 작업 흐름 (시스템 강제)
1. 이슈 받으면 `todo` → `in_progress` → 실행 → 산출물 생성 → `in_review`
2. **비전(감찰관)**이 산출물 검수 → 통과 시 `done` / 미충족 시 `todo` 복귀 + 사유
3. **직접 done 처리 금지** — `in_review`까지만. `done`은 비전만 변경.
4. 차단: `in_progress` → `blocked` + 사유 코멘트
5. done인 이슈를 다시 열지 않는다. 남의 이슈 상태를 변경하지 않는다 (비전 제외).

## 보고
작업 완료 후 CEO(닉 퓨리)에게 `[보고]` 이슈 생성. 포함: **산출물**(경로/링크), **요약**(한 줄), **다음 조치**.

## 핸드오프
```
개발 완료 → "[QA 요청]" → 호크아이
QA 통과  → "[배포 승인]" → 닉 퓨리
QA 실패  → "[버그 수정]" → 스파이더맨
연구 완료 → "[기술 이전]" → 스파이더맨
인력 배치 → "[온보딩]" → 캡틴 아메리카
보안 위반 → "[긴급]" → 닉 퓨리
[보고]   → 닉 퓨리 (CEO)
```

**assigneeAgentId 필수** — 이슈 생성 시 `assigneeAgentId`를 반드시 포함. 미할당 이슈 생성 금지. 위 표에서 담당자의 에이전트 ID를 `에이전트 ID` 테이블에서 찾아 넣어라.

## API 명령 템플릿

```bash
API="http://localhost:3100/api"
CID="af65df00-18ca-4270-9fb0-d4a92d4bba8c"

# 이슈 조회
curl -s "$API/companies/$CID/issues"

# 이슈 생성
curl -s -X POST "$API/companies/$CID/issues" \
  -H 'Content-Type: application/json' \
  -d '{"title":"[접두사] 내용", "assigneeAgentId":"ID", "labelIds":["라벨ID"], "status":"todo"}'

# 상태 변경
curl -s -X PATCH "$API/issues/{issueId}" -H 'Content-Type: application/json' -d '{"status":"in_progress"}'

# 코멘트
curl -s -X POST "$API/issues/{issueId}/comments" -H 'Content-Type: application/json' -d '{"body":"내용"}'
```

**절대 하지 말 것**: Authorization 헤더, API key 생성, 토큰 인증, Settings 변경

## 에러 대응 원칙

**진단 먼저, 재시도는 1회만, 같은 에러 반복 금지.**
1. 에러 발생 → 에러 메시지/traceback 읽고 **원인 분석**
2. 환경 문제 → 1회 재시도. 2회 연속 같은 에러 → 중단
3. 코드 버그 → 재시도 금지. 즉시 이슈 생성 (원인 분석 포함)
4. **절대 금지**: 원인 파악 없는 반복 재시도
